# 🧠 Core Idea

IndexRAG introduces a simple but powerful shift:

> Move reasoning from **inference time** to **indexing time**

It **directly addresses** a core limitation of standard RAG:

> **Independent passage retrieval fails to support cross-document (multi-hop) reasoning**, where evidence is distributed across multiple documents and cannot be retrieved or composed in a single step.

Instead of forcing the model to perform multi-hop reasoning at query time (via iterative retrieval or graph traversal), it **precomputes cross-document reasoning results** and stores them as retrievable units.

# 🧩 Stage 1: AKU & Entity Extraction

- Input: full document (no chunking)
- LLM extracts:
  - QA pairs
  - Entities
- Only **answers** are kept and concatenated into:
```text
AKU = concatenation of answers
```
- Entities are stored at document level

👉 Important:
- QA is only a **scaffold**
- Final unit is **fact-like text (not short answers)**:
  instead of storing short answer spans (e.g., "Henry Edwards"), 
  the system stores self-contained factual statements 
  (e.g., "Aylwin was directed by Henry Edwards"), 
  which preserve relational context and are directly retrievable.

# 🔗 AKU–Entity Binding

There is **no explicit mapping** between QA pairs and entities.
Instead:
1. Document-level binding:
   ```
   (doc) → (AKU, Entities)
   ```

2. At usage time:
   - Use **string matching** to extract:
   ```
   ai[e] = sentences in AKU containing entity e
   ```
👉 Entity acts as a **join key**
👉 As a result:
- Each answer is effectively a **fact-like proposition**
- The AKU becomes:
    > a **collection of atomic factual statements grouped into a single retrieval unit**

# 🔥 Stage 2: Bridging Fact Generation

For entities appearing in multiple documents:
1. Group documents by entity
2. Extract relevant spans (ai[e])
3. Use LLM to generate:
```text
Bridging facts = cross-document reasoning results
```
Example:
```
Doc A: Aylwin → directed by Henry Edwards
Doc B: Henry Edwards → born in Weston-super-Mare

→ Bridging:
"The director of Aylwin was born in Weston-super-Mare"
```

👉 These facts:
- Do not exist in original data
- Encode **multi-hop reasoning**

# 🧠 Retrieval Phase

- Retrieve top-k from:
  - AKUs
  - Bridging facts
- **Balanced Context Selection**
	- Problem:
		- Bridging facts are short → dominate retrieval
		- Short texts are more "concentrated" in the embedding space and are more likely to have a high similarity match with the query → ranking higher.
	- Solution:
		- Limit number of bridging facts (kb)

# ⚡ Key Insight

> Convert multi-hop reasoning into **single-hop retrieval**

# 🧪 Experimental Findings

👉 **Insight:**
> Retrieval quality matters more than coverage (Whether the retrieved information is useful is more important than whether it is abundant or complete.)

👉 **Works best for:**
- Sequential reasoning
👉 **Weak for:**
- Parallel comparison tasks

# ⚠️ Limitations

## 1. Loss of Question Semantics

- Although answers are rewritten into self-contained factual statements, 
  the system still loses **query-specific perspectives**.
- A single fact-like statement encodes only one canonical view of a fact, 
  while different queries may require different projections (e.g., subject-, object-, or relation-focused views).
- As a result, retrieval may be less effective when the query perspective 
  does not align with the stored representation.

## 2. Weak AKU–Entity Alignment

- Uses string matching
- Sensitive to:
  - aliasing (Obama vs Barack Obama)
  - coreference
  - missing mentions

## 3. Bridging Fact Quality

- Generated by LLM
- May contain:
  - hallucination
  - incorrect connections
  - noise

## 4. Entity Extraction Noise

- LLM-based extraction is unstable
- May:
  - miss entities
  - include irrelevant ones

## 5. Scalability Issues

- Entity explosion → too many bridging facts
- Requires careful thresholding

## 6. Loss of Structural Information

- No explicit graph
- Relations are implicit
- Harder to model complex reasoning patterns

## 7. Limited Reasoning Types

- Works well for:
	- Sequential multi-hop
- Struggles with:
	- Parallel reasoning
	- Comparison tasks

# 🚀 Takeaway

IndexRAG shows that:

> If you **materialize reasoning as data**,  
> you can eliminate reasoning at inference time.

This reframes RAG from:
```
retrieve → reason
```
to:
```
pre-reason → retrieve
```
